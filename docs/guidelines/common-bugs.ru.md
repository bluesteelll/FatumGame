# Справочник типичных багов

Это исчерпывающий список багов, встреченных при разработке FatumGame, их первопричин и исправлений. Каждый разработчик обязан прочитать это перед написанием ECS, физического или потокового кода.

---

## Баги Flecs

### World.each() с тегами вызывает краш

**Симптом:** assertion `ecs_field_w_size`, `column_index == -1`.

**Причина:** Передача тега нулевого размера как типизированного параметра `const T&` в `World.each()`. `iterable<>` Flecs сохраняет ссылку, `is_empty_v<const T&>` возвращает `false`, и Flecs пытается обратиться к столбцу данных, который не существует для тегов.

!!! danger "Решение"
    Используйте `query_builder().with<FTag>().build()` + `.each([](flecs::entity E) {...})`. Обращайтесь к тегам через `E.has<T>()` при необходимости. Обратите внимание, что `system<T>().with<Tag>().each()` безопасен, потому что system builder убирает ссылки.

---

### Отложенные операции: 3 случая

#### Случай 1: Между `.run()` системами

**Симптом:** Система B не видит изменения компонентов, сделанные системой A в том же тике.

**Причина:** `.run()` системы не объявляют доступ к компонентам, поэтому Flecs пропускает слияние между ними. `set<T>()` в системе A пишет в отложенный staging; система B читает зафиксированное хранилище.

!!! danger "Решение"
    Используйте буфер подсистемы (`TArray`-член) для передачи данных между системами. Не полагайтесь на компоненты Flecs для передачи данных между `.run()` системами.

#### Случай 2: Внутри одной системы

**Симптом:** `try_get<T>()` возвращает `nullptr` для компонента, только что добавленного через `obtain<T>()` в том же callback.

**Причина:** `obtain()` / `set()` пишут в отложенный staging. `try_get()` / `get()` читают зафиксированное хранилище. Внутри одного callback вновь добавленные компоненты невидимы.

!!! danger "Решение"
    Отслеживайте данные в локальных переменных вместо повторного чтения из Flecs.

#### Случай 3: Теги другой сущности

**Симптом:** Тег, добавленный сущности B внутри системы, итерирующей сущность A, невидим для более поздней системы в этом тике.

**Причина:** `entity.add<FTag>()` для другой сущности откладывается. Если записывающая система не объявляет доступ к FTag, Flecs не выполнит слияние перед следующей системой, запрашивающей FTag.

!!! danger "Решение"
    Выполняйте немедленные побочные эффекты (напр., `SetBodyObjectLayer(DEBRIS)`) вместо полагания на отложенный тег.

    **Реальный пример:** `FragmentationSystem` добавлял `FTagDead` цельной сущности. `DeadEntityCleanupSystem` не видел его до следующего тика. За этот дополнительный тик фрагменты сталкивались с ещё активным цельным телом и взрывались.

---

### Порядок регистрации компонентов Flecs (краш Prefab)

**Симптом:** Краш с `0x80000003` (breakpoint) внутри Flecs при вызове `Prefab.set<T>()`.

**Причина:** Тип компонента `T` не был зарегистрирован через `World.component<T>()` до использования в вызове `Prefab.set<T>()`. Flecs нужны метаданные компонента (размер, выравнивание) до того, как он сможет хранить данные на любой сущности, включая prefab.

!!! danger "Решение"
    Регистрируйте каждый тип компонента через `World.component<T>()` в `RegisterFlecsComponents()` внутри `FlecsArtillerySubsystem_Systems.cpp` ДО того, как любой prefab или система обратится к нему. Это особенно легко забыть при добавлении новых Static-компонентов для новых профилей.

---

### Правила дренирования итератора

#### `.run()` с условиями запроса: ОБЯЗАТЕЛЬНО дренировать

**Симптом:** `ECS_LEAK_DETECTED` в `flecs_stack_fini()` при выходе из PIE.

**Причина:** Система имеет условия запроса, но callback не потребил все совпадения.

!!! danger "Решение"
    Всегда `while (It.next()) { ... }`. Для раннего выхода сначала вызовите `It.fini()`.

#### `.run()` без условий запроса: НЕ финализировать

**Симптом:** Краш в `flecs_query_iter_fini()`.

**Причина:** Двойная финализация. Системы без условий запроса (`system<>("")`) имеют `EcsQueryMatchNothing` -- Flecs автофинализирует после `run()`.

!!! danger "Решение"
    Не вызывайте `It.fini()` или `It.next()` в безпараметровых `.run()` системах.

---

## Баги USTRUCT

### Агрегатная инициализация с GENERATED_BODY()

**Симптом:** Неправильные значения полей, ошибки компиляции или тонкая порча данных.

**Причина:** UHT-генерируемые структуры имеют скрытые члены, внедрённые `GENERATED_BODY()`. Агрегатная инициализация `{value}` присваивает не тому полю.

!!! danger "Решение"
    Всегда используйте именованное присваивание полей:
    ```cpp
    // НЕПРАВИЛЬНО
    entity.set<FItemInstance>({ 5 });

    // ПРАВИЛЬНО
    FItemInstance Instance;
    Instance.Count = 5;
    entity.set<FItemInstance>(Instance);
    ```

---

### TUniquePtr в UHT-генерируемых заголовках

**Симптом:** Ошибка компилятора C4150 ("удаление указателя на неполный тип").

**Причина:** `TUniquePtr<ForwardDeclaredType>` в заголовке UCLASS не работает, потому что UHT-генерируемые файлы `.gen.cpp` инстанциируют конструктор/деструктор, которым нужен полный тип.

!!! danger "Решение"
    Используйте сырой указатель с ручным `new` в `Initialize()` и `delete` в `Deinitialize()`.

---

## Баги физики (Barrage/Jolt)

### Регистрация потока Barrage

**Симптом:** Краш или assertion при обращении к Barrage из рабочего потока Flecs.

**Причина:** Любой поток, обращающийся к Barrage, должен вызвать `GrantClientFeed()` для регистрации в потоковой системе Jolt.

!!! danger "Решение"
    Вызывайте `EnsureBarrageAccess()` в начале любой системы Flecs, обращающейся к Barrage. Использует guard `thread_local` для эффективности.

---

### Самоповреждение / гонка при спавне снаряда

**Симптом:** Снаряд повреждает своего стрелка при спавне.

**Причина:** Физическое тело создаётся и сталкивается ДО полной настройки сущности Flecs. Callback `OnBarrageContact` срабатывает для тела, чей `FlecsId` ещё 0.

!!! danger "Решение"
    В `OnBarrageContact` требуйте `FlecsId != 0` для обоих тел перед обработкой столкновения.

---

### Проверка владельца в системах столкновений

**Симптом:** Снаряд повреждает или отскакивает от своего стрелка.

**Причина:** `DamageCollisionSystem` и `BounceCollisionSystem` не проверяли `OwnerEntityId` у снарядов.

!!! danger "Решение"
    Обе системы должны пропускать столкновения, где `OwnerEntityId` одного тела совпадает с entity ID другого.

---

### Рейкаст прицеливания попадает в снаряды

**Симптом:** Рейкаст прицеливания попадает в собственные снаряды игрока, вызывая хаотичное поведение прицела.

**Причина:** Слой `CAST_QUERY` сталкивается со слоем `PROJECTILE`.

!!! danger "Решение"
    Используйте `FastExcludeObjectLayerFilter({PROJECTILE, ENEMYPROJECTILE, DEBRIS})` для рейкастов прицеливания.

---

### SetPosition ломает траекторию

**Симптом:** Снаряд летит в неправильном направлении после телепортации.

**Причина:** Физическое тело создано в `SimMuzzle` со скоростью, рассчитанной из `SimMuzzle`. Телепортация тела через `SetPosition` без корректировки скорости ломает углы отскока и траекторию.

!!! danger "Решение"
    Никогда не `SetPosition` на снаряде без корректировки скорости для соответствия новой позиции.

---

### FBarragePrimitive::SetPosition -- это ОЧЕРЕДЬ

**Симптом:** Изменение позиции не видно до следующего тика.

**Причина:** `SetPosition()` и `ApplyRotation()` ставятся в очередь `ThreadAcc.Queue` и применяются при следующем `DrainCommandQueue`.

!!! warning "Решение"
    Когда позиция должна быть зафиксирована немедленно (напр., перед `bAutoDetectAnchor` констрейнта), используйте `SetBodyPositionDirect()` или `SetBodyRotationDirect()`, которые обращаются к `body_interface` напрямую.

---

### MoveKinematicBody vs SetBodyPositionDirect

**Симптом:** Тело-якорь констрейнта отстаёт от целевой позиции.

**Причина:** `SetBodyPositionDirect()` телепортирует тело, но оставляет скорость нулевой. Солвер констрейнтов Jolt применяет только стабилизацию Baumgarte (~5%/тик), вызывая большое отставание.

!!! danger "Решение"
    Используйте `MoveKinematicBody(Key, TargetPos, DT)`, который вызывает `body_interface->MoveKinematic()`. Устанавливает скорость на `(target - current) / dt`, чтобы солвер видел движение и мог его отслеживать.

---

### FromJoltCoordinatesD для позиций, не для сил

**Симптом:** Констрейнты рвутся мгновенно с невозможно большими силами.

**Причина:** `CoordinateUtils::FromJoltCoordinatesD()` умножает на 100 (метры в сантиметры). Силы в Ньютонах нуждаются только в обмене осей (Y <-> Z), НЕ в множителе 100x.

!!! danger "Решение"
    Используйте выделенную конвертацию `JoltForceToUE` (только обмен осей) для сил. Оставьте `FromJoltCoordinatesD` для конвертации позиций.

---

### GetBarrageKeyFromSkeletonKey vs GetShapeRef

**Симптом:** Невозможно найти тело пула по SkeletonKey.

**Причина:** Две разные таблицы поиска:

| API | Таблица поиска | Заполняется |
|-----|---------------|------------|
| `GetBarrageKeyFromSkeletonKey()` | `TranslationMapping` | `BindEntityToBarrage()` |
| `GetShapeRef()` | Отслеживание тел | `CreatePrimitive()` |

Тела пула создаются через `CreatePrimitive` (в отслеживании тел), но НЕ привязываются через `BindEntityToBarrage` (нет в `TranslationMapping`).

!!! danger "Решение"
    Для тел пула используйте `GetShapeRef(Key)->KeyIntoBarrage`.

---

### Статические тела Jolt не имеют MotionProperties

**Симптом:** Краш при `GetMotionProperties()` -- возвращает nullptr.

**Причина:** Тела, созданные как `EMotionType::Static` / `NON_MOVING`, НЕ выделяют `MotionProperties`. Смена на `SetMotionType(Dynamic)` только меняет флаг -- НЕ выделяет MotionProperties ретроактивно.

!!! danger "Решение"
    Создавайте тело как `Dynamic` / `MOVING` с самого начала, если нужен контроль массы, демпфирования или скорости. Используйте `AllowedDOFs = 0x3F` для обхода ограничения RotationY слоя MOVING. Если создание констрейнта в том же `EnqueueCommand` (до `StepWorld`), нет падения под гравитацию на 1 тик.

---

### JPH Non-Copyable Filter

**Симптом:** Ошибка компилятора C2280 ("попытка обращения к удалённой функции").

**Причина:** Тернарный оператор с `IgnoreSingleBodyFilter` пытается скопировать некопируемый тип Jolt.

!!! danger "Решение"
    Используйте блоки if/else вместо тернарного оператора при работе с типами фильтров Jolt.

---

### Рандомные пули на близкой дистанции

**Симптом:** Пули летят в случайных направлениях на близкой дистанции.

**Причина:** Параллакс ствола -- позиция дульного среза и точка прицеливания создают расходящийся угол на малых дистанциях.

!!! danger "Решение"
    Установите `MinEngagementDist = 300u` и добавьте проверку безопасности скалярного произведения (если `< 0.85`, используйте направление прицеливания вместо направления от дула к цели).

---

## Баги персонажа и рендеринга

### Фолбек GetMuzzleLocation должен использовать камеру

**Симптом:** Выстрелы идут из неправильной позиции, когда меш оружия недоступен.

**Причина:** Фолбек использовал `GetActorLocation()` (центр капсулы, ~60 единиц ниже камеры) вместо позиции камеры, вызывая параллакс.

!!! danger "Решение"
    Фолбек должен использовать `FollowCamera->GetComponentLocation()`.

---

### Позиция VFX смерти

**Симптом:** VFX смерти спавнится в неправильном месте.

**Причина:** К моменту выполнения `DeadEntityCleanupSystem` физическое тело отскочило от точки столкновения.

!!! danger "Решение"
    Сохраняйте позицию столкновения в `FDeathContactPoint` в момент столкновения. Используйте её в `DeadEntityCleanupSystem` для размещения VFX.

---

### Накопление VelocityScale (двойное масштабирование)

**Симптом:** Игрок неконтролируемо ускоряется при замедлении времени с `bPlayerFullSpeed = true`.

**Причина:** `mLocomotionUpdate = SmoothedH * VelocityScale`. На следующем тике `GetLinearVelocity()` возвращает МАСШТАБИРОВАННУЮ скорость. Сглаживание читает это как `CurH` и снова умножает на `VelocityScale`. Накапливается каждый кадр.

!!! danger "Решение"
    Перед сглаживанием движения отмените предыдущее масштабирование:
    ```cpp
    if (VelocityScale > 1.001f)
    {
        CurH *= (1.f / VelocityScale);
    }
    ```
    То же применимо к любому считыванию `GetLinearVelocity()`, используемому для проверок скорости (напр., скорость входа в скольжение).

## Баги UI и виджетов

### TTripleBuffer: Write vs WriteAndSwap

**Симптом:** UI никогда не обновляется несмотря на запись новых данных.

**Причина:** `Write()` не устанавливает флаг dirty. Только `WriteAndSwap()` публикует новые данные на сторону чтения.

!!! danger "Решение"
    Всегда используйте `WriteAndSwap()`, никогда голый `Write()`.

---

### Осиротевший ISM при Tombstone

**Симптом:** Невидимое столкновение с фантомным объектом; ISM-меш остаётся после уничтожения сущности.

**Причина:** `TombstonePrimitive()` уничтожил физическое тело без удаления соответствующего ISM-экземпляра.

!!! danger "Решение"
    Очистка ISM добавлена в `TombstonePrimitive()` для типов `SFIX_BAR_PRIM` и `SFIX_GUN_SHOT`.

---

## Баги жизненного цикла

### Гонка деинициализации (Use-After-Free)

**Симптом:** Краш при выходе из PIE внутри Flecs `progress()` или кода Barrage.

**Причина:** `Deinitialize()` game thread уничтожает мир Flecs, пока sim thread ещё внутри `progress()`.

!!! danger "Решение"
    Используйте атомарные барьеры: `bDeinitializing` + `bInArtilleryTick`. Deinitialize устанавливает флаг и ожидает в spin-wait до выхода sim thread из текущего тика.

---

### Гонка SelfPtr

**Симптом:** Краш в рабочем потоке Flecs при обращении к `UBarrageDispatch::SelfPtr` во время завершения.

**Причина:** Deinitialize обнуляет `SelfPtr`, пока рабочий поток находится внутри системы.

!!! danger "Решение"
    Кешируйте указатель как `CachedBarrageDispatch` при регистрации систем. Очищайте после остановки sim thread.
